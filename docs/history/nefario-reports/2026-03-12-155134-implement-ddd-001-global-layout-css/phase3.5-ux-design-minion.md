- [ux-design]: The `padding-inline` from `main > .section > div` also applies to `.default-content-wrapper`, which matches the same selector. This double-applies horizontal padding inside the reading column, reducing actual text width to approximately 486–548px rather than the DDD-specified 550–612px (68ch minus 64px padding at desktop).
  SCOPE: `styles/styles.css` — Task 3, Phase C, inner-tier rule
  CHANGE: Add `padding-inline: 0` to `main > .section > .default-content-wrapper` so the reading column is not padded inward. The outer-tier padding belongs on block wrappers and full-width containers, not on the narrow reading column whose centering is already handled by `margin-inline: auto` and `max-width: var(--measure)`.
  WHY: `default-content-wrapper` is a `div` and matches both the `main > .section > div` selector (which sets `padding-inline`) and its own selector (which sets `max-width: var(--measure)`). CSS cascade leaves the `padding-inline` in place. At desktop the result is 68ch outer width but only ~52ch of actual text — narrower than comfortable reading width and narrower than the DDD's stated geometry. The drafts test page will expose this at 900px+ but it is not obvious without measuring.
  TASK: Task 3

- [ux-design]: Open Question #5 from DDD-001 (focus ring contrast) is unresolved and is not addressed by any task in this plan. `--color-accent` (#D9B84A) against `--color-background` (#F6F4EE) yields approximately 2.7:1 contrast — below the WCAG 2.4.13 minimum of 3:1 for focus indicators. No task checks or updates the focus ring color.
  SCOPE: `styles/styles.css` — button focus rule, any `:focus-visible` declarations
  CHANGE: Before shipping, verify what color is currently used for focus rings in `styles.css`. If `--color-accent` is used or if there is no explicit focus indicator, add a `:focus-visible` rule that uses `--color-heading` (#3F5232) or `--color-text` (#3A3A33) as the focus outline color, both of which achieve sufficient contrast against the warm white background. This is a design-phase fix, not an accessibility audit.
  WHY: The DDD explicitly flags this gap ("The implementing agent must verify focus indicator contrast before shipping"). This plan ships without addressing it. Focus rings are the primary keyboard navigation affordance — insufficient contrast fails WCAG 2.4.13 (AA) and makes the site inaccessible to keyboard users.
  TASK: Task 3 (or a new Task 3b before the approval gate)
